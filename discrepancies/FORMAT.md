# Discrepancy report format

Each Sonnet agent audits ONE Rust port file with a **three-way** comparison and
writes one `discrepancies/<name>.md` file in the structure below.

## Sources compared

- **Rust port** (under audit) — `src/codemp/game/<name>.rs`
- **PC oracle** (source of truth) — `oracle/codemp/game/<Name>.c`
- **Xbox grayj** (old origin) — cloned separately (see workflow); 82/86 files have one.
  4 files are PC-only (no Xbox counterpart): `npc_ai_galakmech`, `npc_ai_interrogator`,
  `npc_ai_mark1`, `npc_ai_mark2` — those get a two-way Rust-vs-PC comparison only.

## Classification (the `Class` column)

A finding is any Rust body that **diverges in behavior from the PC oracle**. It is
then attributed by what the Rust *matches*:

| Rust vs PC oracle | Rust vs Xbox grayj | Class |
|---|---|---|
| differs | **matches** | `xbox-residue` — the port followed old Xbox logic (issue #1's target) |
| differs | differs | `port-bug` — novel translation error / incomplete stub |
| differs | deliberate, behavior-preserving | `intentional` (rare) |
| differs | can't attribute cleanly | `unsure` |
| matches | — | not a finding |

## Markdown template

```markdown
# Discrepancies: src/codemp/game/<name>.rs

- **Rust:** src/codemp/game/<name>.rs
- **PC oracle:** oracle/codemp/game/<Name>.c
- **Xbox grayj:** <path or "none (PC-only file)">
- **Verdict:** <one line — e.g. "Clean", "N divergences (M xbox-residue)">

## Findings

| Function | Rust line(s) | PC line(s) | Xbox line(s) | First divergence | Class |
|----------|--------------|-----------|--------------|------------------|-------|
| SP_misc_model | 412–448 | 388–420 | 401–433 | `g_misc.rs:419 — if health > 0 {` | xbox-residue |

### Detail
<2–4 lines per xbox-residue / unsure row: Rust snippet, PC snippet, Xbox snippet,
so a human can confirm the attribution. port-bug rows stay table-only.>
```

### Column rules

- **First divergence** points AT the suspected first point of divergence from the PC
  oracle (a `file:line` + minimal snippet). It does NOT describe what differs.
- **Xbox line(s)** = where the corresponding code sits in grayj (`n/a` if no Xbox file,
  `absent` if the logic doesn't exist there).
- A row may be classed `xbox-residue` ONLY if the agent verified the Rust matches the
  Xbox version at the cited Xbox line(s).
- Clean files still get a file with the header and "No behavioral discrepancies found."

## Provenance note

The first 10 reports in this folder (`w_saber`, `bg_pmove`, `ai_wpnav`, `w_force`,
`ai_main`, `q_math`, `g_combat`, `g_client`, `g_items`, `bg_saber`) were generated by an
EARLIER **two-way** prompt (Rust vs PC only) — their `Class` column used
`likely-bug` instead of `port-bug`/`xbox-residue`, and they have no Xbox column.
They should be regenerated with the three-way workflow for consistent attribution.
